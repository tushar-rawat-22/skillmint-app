# SkillMint Premium Visual Direction

> **Status: Historical visual direction.** Its dark-cockpit guidance is superseded by the current light-first system in [PREMIUM_UX_DIRECTION.md](../PREMIUM_UX_DIRECTION.md) and [AGENTS.md](../../AGENTS.md).

## Purpose

Sprint 7.10 exists to make SkillMint feel beta-ready from a trust and founder-appeal standpoint.

The product should feel like a serious career cockpit: minimal, dark, polished, and useful without becoming an admin dashboard or a flashy game.

## Visual Positioning

- Premium dark career operating system
- Clean SaaS clarity with light gamified progress
- TradingView/Bloomberg-inspired intelligence density where it helps decisions
- Linear/Raycast-style restraint in spacing, surfaces, and interactions
- Strong emerald identity, used intentionally instead of everywhere

## Design Principles

- Fewer but stronger cards
- Deep near-black backgrounds with subtle radial depth
- Glassy dark surfaces with thin borders and restrained shadows
- Strong headings with calm explanatory copy
- Clear primary actions, softer secondary actions
- Semantic color: emerald for growth, amber for warning, red for risk, blue/violet for analysis and future path
- No fake metrics, testimonials, urgency, or paid entitlements

## What Changed In Sprint 7.10

- Landing hero, product preview, problem, loop, feature, and CTA sections received stronger premium depth.
- Dashboard shell, sidebar, topbar, command center cards, gauges, metric strip, missions, and feedback widget were visually unified.
- Setup, upload, resume, ATS, roadmap, profile, and settings received more consistent dark cockpit surfaces.
- Upload now feels like the first serious product action rather than a plain file input.
- Upgrade-interest cards were softened so they feel like beta signal capture, not a hard sale.

## Sprint 7.10B Cohesion Fixes

- Landing copy and the four-step loop were tightened so the page remains premium without dead space.
- Login and signup must visually match the landing theme.
- ATS, resume, profile, and settings must feel like part of the same product system.
- Shareable snapshots must never show weak operational tasks like "Upload a resume" as a candidate improvement.
- Profile identity uses initials only during beta. Do not add real image upload until account/profile scope expands.

## What Not To Add

- No payment provider code during beta freeze polish.
- No fake success claims, fake testimonials, or fake user numbers.
- No new scoring, roadmap, ATS, parser, extraction, or schema behavior just to improve visuals.
- No decorative clutter that does not explain readiness, proof, match, or next action.

## Beta Freeze Visual Acceptance

Before tagging beta freeze, the founder should be comfortable showing:

- The landing page as the public product story.
- Login/signup as part of the same premium product experience.
- The dashboard as the career command center.
- Setup and upload as the first-run flow.
- Resume, ATS, and roadmap as premium intelligence outputs.
- A shareable snapshot that is truthful, non-embarrassing, and ready only after resume analysis.

Founder visual approval is required. If the product feels technically correct but emotionally flat, beta freeze should wait.
